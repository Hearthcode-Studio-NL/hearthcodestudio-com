> Per [HearthCode-Vault/04-Standards/Testing-Policy § Accessibility][tp], every HearthCode release of a UI project must pass a manual keyboard + screen-reader walkthrough on the top-3 user flows. This is the v1 launch signoff for the studio's own marketing site.

## Scope

This is a single-page long-scroll marketing site with two short Dutch-language legal pages. The "top-3 flows" are intentionally small — the site has no forms, accounts, payments, or transactional surfaces.

**Flows tested:**

1. **Discover the studio.** Visitor lands on `/`, reads the Hero → Approach → Person → Work → Contact sections in order via scroll/Tab, locates the email CTA.
2. **Reach the studio.** Visitor activates the "Start a conversation" CTA (mailto link) or the explicit `info@hearthcodestudio.com` link in the Contact section.
3. **Read legal context.** Visitor navigates from the footer to `/privacy` and `/toegankelijkheidsverklaring`, reads each, returns to home via the header logo or browser back.

**Out of scope:**

- Any flow that doesn't yet exist on the site — there is no client portal, no sign-up, no contact form, no analytics, no cookie banner. The next signoff (presumably `0002-…`) will broaden scope when more surfaces ship.

The SEO target was lowered from `1.0` to `0.6` for the `v0.1.0` release per [ADR-0003][adr3]. Reason: `src/middleware.ts` emits `X-Robots-Tag: noindex, nofollow` on every host that is not `hearthcodestudio.com`, including `localhost` (which CI tests against). Lighthouse penalises noindex pages in the SEO category. The threshold ratchets back to `1.0` at DNS cutover.

LCP appears as a `warn` in CI runs (one cold-start outlier hit ~9s; other runs were 3.0–3.6s) but does not block — the binding gate is `categories:performance` overall, not the per-metric LCP.
